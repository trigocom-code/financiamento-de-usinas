● Catálogo honesto · vivo × estático

Nem todo material é "vivo". Aqui eles estão separados pelo que realmente são: os que se atualizam sozinhos (fetch de JSON canônico a cada visita) e os que são editados à mão.

Vivos têm selo de frescor: verde = últimas 24h · amarelo = 1 a 7 dias · vermelho = mais de 7 dias (peça regeneração). Referência = estático, o carimbo mostra a data da última edição real — sem alarme de defasagem.

Como o frescor é medido. Cada cartão consulta a API pública do GitHub (`api.github.com/repos/.../commits?path=…`) ao abrir e mostra o timestamp do último commit que tocou aquele arquivo. Nos vivos, é a última regeneração pelo monitor. Fontes do monitor: F1 (páginas oficiais/DOU), F2 (imprensa setorial) e F3 — associações que entregam ponteiro por e-mail (hoje: ABSOLAR, via seus Grupos de Trabalho). F3 não fecha um fato em VERDE sozinha quando a matéria exige F1.

3ª Camada · Os 8 vetores SOA/SOS — discriminados

Cada vetor é uma linha de serviço com cliente-tipo próprio (Matriz Cliente × Vetor: A·Geração · B·Transmissão · C·Consumidor · D·Banca · E·Institucional · F·Seguros). Os folders da 1ª camada são as vitrines; os monitores da 2ª camada são os gatilhos; o Motor QSV da 4ª camada é a prova.

1ª · Folders (vitrine)→ 2ª · Monitores (gatilho)→ 3ª · Vetores (linhas de serviço)→ 4ª · Motor QSV (prova)

4ª Camada · Motor QSV — a base de prova dos folders

A camada que liga os dados oficiais aos folders com prova auditável por terceiro (na arquitetura técnica, é a "Camada 3": dados → folders → prova). Um único motor, um manifesto por folder — nenhum número chega ao folder sem passar pelo QSV, o diferencial da SOA/SOS.

Pipeline de prova

Como o motor produz a evidência

① Ingestão (SQL versionado, read-only) → C1 hash → C2 timestamp → C3 gêmeo digital (pvlib) → C4 vizinhos (H3) → Σ score 0–3 → Evidence Pack → C5 batch on-chain (Polygon) → Feed → Folder. Degradação graciosa: qsvDegraded, nunca número sem lastro.
off-chain first · 5 TX/dia
Auditável por terceiro

Verificação independente

De posse do Evidence Pack, o auditor re-executa o SQL, confere o input hash, valida o Merkle proof e checa a root on-chain — reproduzindo o número bit a bit, sem depender da SOA.
SQL + Merkle + on-chain
